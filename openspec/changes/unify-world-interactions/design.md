## Context

出生房间前门当前自行连接 `MouseEntered`、`InputEvent` 和玩家 `ArrivedAtTarget`。这种实现无法安全支持多个同时存在的交互对象，也把地点切换写死在房间脚本中。世界物品将引入另一套右键逻辑，必须建立统一优先级和生命周期。

## Goals / Non-Goals

**Goals:**
- 统一世界对象交互契约和走近后执行流程。
- 保证单一待处理目标、可取消和执行前复核。
- 抽离地点加载与玩家出生点定位。

**Non-Goals:**
- 不实现任务编辑器、对话树或复杂交互轮盘。
- 不负责物品库存规则和厉鬼 AI。

## Decisions

1. **使用抽象基类或接口表达交互能力。** 交互对象提供提示、距离、交互点和执行方法；协调器不识别具体门或物品类型。
2. **玩家拥有唯一 `InteractionController`。** 控制器保存待执行对象并独占到达回调，新的交互请求替换旧请求。
3. **执行前二次校验。** 自动寻路结束后重新检查对象有效性、距离和可用性，避免对象被删除或状态改变。
4. **地点切换由 `Main` 下的服务统一管理。** 地点通过 ID 注册 PackedScene；出生点使用稳定 `SpawnId`，玩家/HUD/背包保持常驻。
5. **输入优先级为 UI > 世界交互 > 鬼域/空地使用。** 成功接收请求的层必须标记输入已处理。

## Risks / Trade-offs

- [C# 接口无法直接导出 Godot 信号] → 交互基类负责信号和节点生命周期，接口只表达纯行为，实施时择一保持一致。
- [目标在寻路中被释放] → 使用弱引用或 `GodotObject.IsInstanceValid` 检查，失效时取消。
- [地点切换时导航地图尚未同步] → 出生点直接定位；需要自动寻路的交互等待新地点就绪。
- [不同鼠标按钮语义冲突] → 交互对象声明触发按钮或由协调器统一映射，默认左键门/调查、右键拾取。

## Migration Plan

先实现框架和地点服务，再迁移前门，最后迁移世界物品。迁移期间保留旧方法但不得同时连接输入；验证后删除点对点转场方法。
