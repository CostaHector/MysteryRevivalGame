## Context

前四个 Change 分别提供库存、鬼域、交互和威胁能力，但仍需要内容层将它们组合为可玩的首个副本。当前地点只有出生房间具有导航，废弃老宅仅有背景。副本流程必须保持轻量，避免过早构建通用任务编辑器。

## Goals / Non-Goals

**Goals:**
- 提供从开始、探索、遭遇到成功或失败的短篇闭环。
- 以事件驱动阶段控制地点、目标、世界标记和检查点。
- 为后续章节复用最小的场景进度接口。

**Non-Goals:**
- 不实现完整对话系统、开放世界任务日志或跨启动存档。
- 不覆盖原著全部“七种敲门鬼”内容。

## Decisions

1. **副本使用显式阶段枚举和目标集合。** 每阶段声明进入动作、完成条件和下一阶段，避免散落布尔值。
2. **通过领域事件推进。** 拾取、地点进入、调查完成、鬼被压制和出口交互触发进度，不在 `_Process` 中轮询。
3. **地点场景保留编辑器数据。** 导航多边形、碰撞区、出生点和背景属于真实场景数据；行为节点仍按项目习惯由 C# 创建。
4. **出口验证使用世界标记和关键物品。** 玩家必须获得线索或完成反制，错误出口提供可读失败反馈而非无提示阻挡。
5. **检查点为内存快照。** 阶段进入时由场景控制器请求遭遇/库存系统创建快照；持久化留待后续 Change。
6. **结算独立于失败 UI。** 成功界面显示关键结果并提供重玩或返回标题，不直接退出应用。

## Risks / Trade-offs

- [跨 Change API 尚未完全落地] → 规格依赖事件与接口语义，不绑定具体类名；实施顺序固定在前四项之后。
- [内容不足导致流程过短] → 优先保证目标可理解、威胁有反制和结局完整，再扩充支线调查。
- [场景数据与代码优先约定冲突] → 只在 `.tscn` 保存导航、碰撞、背景和标记点，动态行为仍由代码创建。
- [阶段重置遗漏世界对象] → 地点加载时根据世界标记重新构建一次性物品和交互状态。

## Migration Plan

先扩充地点数据，再建立无威胁的阶段推进，之后接入世界物品和脚步鬼，最后加入出口验证、结算和重玩。每个阶段可独立通过开发命令跳转验证。

## Open Questions

- 首版最终出口采用“获得手机线索并成功压制一次脚步鬼后可识别真门”的规则，后续试玩可调整叙事包装。
