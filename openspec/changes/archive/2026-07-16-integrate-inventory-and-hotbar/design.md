## Context

仓库已有 `ItemData`、`ItemDatabase`、`ItemStack` 和 `InventoryModel`，但没有全局所有者，背包与快捷栏也没有订阅模型。当前 UI 分别自行构造格子并硬编码内容，无法保证同一物品在两处一致显示。项目采用代码优先构建节点、`Main` 作为运行时父节点、Godot C# 信号驱动更新。

## Goals / Non-Goals

**Goals:**
- 建立唯一库存数据源和稳定的槽位索引规则。
- 让背包、快捷栏、拾取和使用都通过模型 API 读写数据。
- 支持现有技术方案中的堆叠、消耗、tooltip 和点击移动行为。
- 为后续物品效果、任务道具和存档提供可扩展边界。

**Non-Goals:**
- 本 Change 不实现具体灵异物品的复杂效果。
- 不实现拖拽、拆分堆叠、丢弃、商店和持久化存档。
- 不重构所有世界交互；世界物品只提供可工作的最小拾取入口，后续由统一交互 Change 迁移。

## Decisions

1. **使用 `GameState` Autoload 持有库存。** 库存生命周期跨地点存在，不应挂在会被替换的房间节点下。相比通过 `Main` 字符串查找，Autoload 可减少耦合并为存档提供固定入口。
2. **36 格库存为唯一数据源，快捷栏映射索引 27–35。** 不维护第二份快捷栏数组，避免同步和移动时出现双写问题。
3. **模型通过 `SlotChanged(index)` 发出细粒度更新。** UI 只刷新受影响槽位；初始化时执行一次全量刷新。相比每帧轮询，这更符合 Godot 信号模式。
4. **槽位内部统一使用 0-based 索引。** 节点名称可继续为用户可读的 `Slot1`，但业务索引全部为 0–35，消除当前 UI 与模型的偏移风险。
5. **使用请求由服务层协调。** UI 只提交槽位索引；服务先执行效果，再根据结果消耗耐久或数量。当前无效果物品可直接视为成功消耗。
6. **世界物品只在完整数量入包后销毁。** `AddItem` 返回剩余数量；若存在剩余，世界节点保留剩余堆叠，避免背包满时丢失物品。
7. **背包点击采用单一选中索引状态机。** 点击空槽仅在已有来源槽时移动；点击非空槽改选；再次点击当前槽取消；关闭背包清空选择。

## Risks / Trade-offs

- [现有 `ItemStack` 的单件耐久语义不足以表达同堆不同耐久] → 初期只允许相同耐久堆叠，后续需要拆分堆栈时再扩展。
- [Autoload 初始化顺序错误会导致 UI 空引用] → 在 `_EnterTree()` 创建库存，并让 UI 在 `_Ready()` 后绑定。
- [右键同时触发拾取与使用] → 世界物品在处理成功时标记输入已处理；快捷栏使用只响应未处理的右键空地事件。
- [大量 UI 节点通过名称查找脆弱] → 创建格子时保存视图引用数组，不在刷新路径中反复字符串查找。

## Migration Plan

1. 注册 `GameState` 并完成库存模型缺失的边界校验。
2. 将快捷栏和背包改为模型驱动，保留现有视觉布局。
3. 在 `Main.NewGame()` 初始化测试物品，验证两处同步。
4. 接入使用、拾取、tooltip 和移动交互。
5. 删除硬编码图标和固定数量文本。若迁移失败，可暂时恢复静态 UI，但保留模型代码不会破坏场景加载。
