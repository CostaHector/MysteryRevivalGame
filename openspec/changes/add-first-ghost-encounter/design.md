## Context

游戏尚无敌人和真实状态。第一只鬼必须以规则制造压力，而不是复制传统追踪怪物。现有顶部 HUD 可作为状态展示入口，库存和鬼域未来可通过统一压制接口参与反制。

## Goals / Non-Goals

**Goals:**
- 建立玩家生存状态与失败流程。
- 实现一只可读、可测试、有反制窗口的脚步鬼。
- 让遭遇可从检查点重开。

**Non-Goals:**
- 不实现通用行为树、多鬼协作、武器战斗或永久存档。
- 不要求首版具有完整角色动画和高成本美术。

## Decisions

1. **玩家状态使用独立模型。** 身体、精神、复苏和死亡由 `PlayerSurvivalState` 持有，HUD 订阅信号，厉鬼不得直接修改文本。
2. **脚步鬼使用显式有限状态机。** `Dormant → Marked → Hunting → Attacking`，压制进入 `Suppressed` 后根据规则恢复。
3. **杀人规律由遭遇控制器触发。** 黑暗区域、停留时间或剧情标志组合成触发条件，鬼本身只消费触发事件。
4. **追猎先以距离曲线和音频表现为主。** 无需持续显示鬼的精灵；脚步声间隔、方向和压迫反馈表达接近程度。
5. **反制使用 `Suppress(duration, source)` 接口。** 红鬼烛和鬼域都可调用，避免厉鬼依赖具体物品类型。
6. **死亡流程冻结当前遭遇并显示覆盖 UI。** 重开恢复检查点快照，不重新创建整个应用。

## Risks / Trade-offs

- [纯音频威胁难以调试] → 提供开发模式状态标签和距离日志，发行模式隐藏。
- [杀人规律不清晰导致随机死亡感] → 在首次触发前提供环境和声音预警，并保证固定最短反应时间。
- [反制接口先于具体物品效果] → 提供测试调用和事件，库存 Change 完成后再接真实物品。
- [重开残留信号或计时器] → 遭遇控制器统一创建、停止和重置计时器，避免节点自行泄漏状态。

## Migration Plan

先接入玩家状态和 HUD，再实现无伤害的开发状态机，随后启用袭击与失败 UI，最后连接压制来源和检查点重开。

## Open Questions

- 首版触发条件采用“进入黑暗区域并停留”作为确定性规则，具体时间在试玩中调优。
